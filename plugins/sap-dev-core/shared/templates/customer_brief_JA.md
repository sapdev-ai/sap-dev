# Customer Project Brief — 1ページフォーム

これは、AI が高品質で配備可能な ABAP を生成するために、**お客様がプロジェクト
に対して 1 度だけ提供する必要がある最小限の情報** です。設計書を置き換える
ものではありません — 個々の設計書をまたいで共有される **プロジェクト全体の
コンテキスト** を生成器に与えるものです。

このフォームを、お客様のプロジェクトリーダーまたはビジネスアナリストに
お渡しください。多くの項目は 10 分程度で記入でき、残りは「TBD — Basis /
セキュリティチームに確認」とマークして後で記入できます。**新しいテンプレート
を覚える必要はありません — このブリーフは、お客様が現在使っているドキュメント
形式と並んで存在します。**

> 💡 **記入例として
> [`customer_brief_sample_JA.md`](customer_brief_sample_JA.md)
> を参照してください。** 仕様書の構造リファレンスは
> [`spec_template_JA.xlsx`](spec_template_JA.xlsx) です。
> 白紙のフォームを最初から記入するより、サンプルをコピーして値を差し替える
> ほうが簡単です。

## 6. 品質基準

| 項目 | 選択 |
|---|---|
| ABAP Unit テストは必須? | `必須` / `あれば良い` / `不要` |
| ATC は通過必須? | `必須 (優先度 1+2 がゲート)` / `優先度 1 のみ` / `不要` |

| メソッドの最大行数? | `50` (既定) / `100` / 上限なし |

## 7. 設計書ごとの契約 (各設計書で繰り返す小さなフォーム)

新しい設計書を引き渡される際は、**お客様の既存ドキュメント内に以下の 4 つ
の枠付きセクション** を含めてもらうだけで十分です (テンプレート変更は不要):

1. **インターフェイス契約** — 入力 / 出力 / 例外 (型付きが望ましい)
2. **検証ルール** — 番号付きで、各ルールが `<MSG_CLASS>-<MSGNO>` を指す
3. **処理フロー** — 番号付きステップ、各ステップは動詞 1 つ
4. **DDIC オブジェクト** — 名前 / 型 / 桁数 / データ要素の表

これら 4 つは、AI がコードを推測せずに生成するために必要な最低限です。
「フォーマットはそのまま、4 つの枠を追加するだけ」と説明して導入してくだ
さい。

---

## 8. 設計書ごとのテストケース (ゴールデン I/O)

検証ルールごとに、「入力 X が与えられた場合、出力 Y / エラー Z を期待」と
いう 2〜3 行を提供してください。ビジネスアナリストはこれらを既に頭の中に
持っています。フォーマットの提案:

```
ルール 3: 登録区分は I または U のみ
  - 入力: 登録区分=A → ERROR ZHKMSG01-001
  - 入力: 登録区分=I → OK
  - 入力: 登録区分=U → OK
```

## 各スキルでの使用方法

- `/sap-docs-extract` は設計書 + ブリーフを読み、`_PGM_summary.txt`、
  `_process.txt` 等を出力します。
- `/sap-gen-abap` はブリーフを使って次を選びます: 最新 vs クラシック構文、
  OOP vs FORM スキャフォールド、DDIC 型 vs 基本型、パフォ帯 → SQL パターン。
- `/sap-check-abap` はブリーフの品質基準で、どの指摘が配備をブロックするか
  を判定します。
- ブリーフは `{custom_url}\customer_brief.md` (プロジェクトごとの上書き) に
  配置します。デフォルトは
  `<SAP_DEV_CORE_SHARED_DIR>/templates/customer_brief.md` (英語) または
  `customer_brief_JA.md` (日本語、本ファイル) を使用します。
